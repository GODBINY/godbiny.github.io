##### 서버가 클라이언트 인증을 확인하는 방법
- cookie, session, token 방식이 존재한다.

##### Cookie
- 특징
	- client(browser)에 key-value 형태의 작은 정보를 저장한다.
	- 사용자가 웹사이트에 접속하면 서버가 쿠키 정보를 브라우저에 설치
	- 이후 요청마다 요청 헤더에 쿠키를 포함시켜서 전송한다.
- 장점
	- 사용자 식별을 간단하게 할 수 있다.
	- 각 브라우저별로 개별 관리가 가능하다.
- 단점
	- 보안 문제
		- 쿠키 값이 그대로 노출되어있어 탈취/조작 위험 존재
	- 용량 제한
		- 쿠키 크기는 제한이 있음.
	- 브라우저 호환성 이슈
		- 각 브라우저마다 동작 방식이 다름.
	- 네트워크 부하
		- 쿠키가 커질 수록 요청할 때마다 네트워크 부하가 발생한다.
##### Session
- 특징
	- 민감한 인증 정보를 브라우저가 아닌 서버 측(메모리, 로컬 파일, DB)에 저장한다.
	- 사용자가 로그인을 하면 서버에 session 객체가 생성된다.
	- 이후 session id만 브라우저 쿠키에 저장하여 요청시마다 전송한다.
- 장점
	- 쿠키에 비해 강화된 보안
		- 쿠키에 세션ID만 담기므로 개인정보 노출 우려가 적음
- 단점
	- 세션 탈취 위험
		- 세션 ID만 가지고서도 사용자인척 할 수 있음.
	- 서버 부하
		- 세션을 서버에 저장하므로 트래픽 증가시 서버 부하가 커진다.
	- 확장성
		- 사용자가 많아지면 서버 확장 및 관리가 어렵고 성능이 저하된다.
##### Token
- 특징
	- 클라이언트가 서버에 인증되면 토큰을 발급받아 클라이언트(쿠키, 로컬스토리지 등)에 저장한다.
	- 이후 요청마다 헤더에 토큰을 담아 서버로 전송한다.
	- 서버에서는 토큰 자체를 검증하여 인증한다.
- 장점
	- 서버 확장성
		- 상태를 유지하지 않는 stateless 방식으로 서버 부하가 적고 확장성이 좋다.
	- 저장소 불필요
		- 별도의 세션 저장소가 필요 없음.
	- 위변조 방지
		- Signature 검증으로 토큰 위조를 방지한다.
	- 크로스 플랫폼 호환성
		- 모바일, 외부 서비스, OAuth 등 다양한 환경을 지원한다.
- 단점
	- 토큰 길이 증가
		- 정보가 많아질수록 토큰이 길어져 네트워크 부하가 발생한다.
	- Payload 노출
		- 토큰의 Payload가 Base64 인코딩으로 복호화가 가능하므로 중요 정보 노출 위험이 있다.
	- 탈취 대처 어려움
		- 토큰을 탈취당하면 만료 전까지는 누구나 접근 가능하다.
	- 보안 강화 대책 필요
		- 짧은 유효기간을 두거나, Refresh Token을 두는 등 추가 보안 장치가 필요하다.
